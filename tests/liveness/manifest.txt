# vui_overread: a synthetic SPS whose VUI over-reads past the SPS rbsp (the SPS
# NAL's last 2 bytes are trimmed after generation, mimicking a common encoder bug
# ffmpeg reports as "Overread VUI by N bits" and tolerates). The strict
# rbsp_trailing_bits check used to reject the whole SPS with EBADMSG, dropping the
# entire stream (0 frames). Since the VUI is the last, non-normative element and
# every decoding-relevant field before it is already parsed and bounds-checked,
# the fix accepts the SPS (reverting the VUI's two values to the inferred
# defaults) and decodes the IDR (1 frame). Real captures hit this (a Main and a
# High clip in a sweep, both flagged "Overread VUI by 8 bits" by ffmpeg too).
# Regression (VUI-overread SPS tolerance) => EBADMSG / 0 frames.
vui_overread 1
# cabac_overread: 24 single-MB I_PCM pictures (synthetic CABAC, gen_cabac_overread.py).
# Each slice decodes its whole picture, but raw 0xFF bytes after the PCM samples leave the
# re-initialised cabac engine with offset >= range (end_of_slice = 1) AND a non-zero
# msb_cache - the look-ahead state a dense final slice leaves when its data fills the NAL.
# The strict cabac end check then tripped EBADMSG on these COMPLETE slices, so their
# remaining_mbs was never zeroed and the pictures never finalized. They pile up undelivered
# until the DPB overflows and the decoder spins ENOBUFS mid-stream (before end-of-stream, so
# the EOS forward-progress valve cannot mask it). A correct decoder ignores the look-ahead
# slop on a slice that decoded every macroblock and emits all 24. ffmpeg decodes them too.
# Regression (cabac end-of-slice over-read tolerance) => mid-stream stall.
cabac_overread 24
# cabac_orphan: an IDR then 23 single-MB-per-picture frames (synthetic CABAC,
# gen_cabac_orphan.py), where picture 1 codes only 1 of its 2 macroblocks (a slice
# that ends via end_of_slice mid-picture) so it never finalizes - the state a
# corrupt stream leaves when a slice errors mid-frame. The SPS sets
# max_num_reorder_frames = 0 so complete pictures are output immediately; the held
# incomplete picture (lowest POC, skipped by get_frame mid-stream) is bumped into
# the 16-entry queue and then shifted out by the later bumps without being
# delivered. Orphaned (in to_get_frames but absent from the queue), it made
# bump_all_frames spin ENOBUFS forever at end-of-stream, so the decoder delivered
# 23 of 24 and stalled. The fix finalizes and re-queues the orphan so all 24 come
# out (ffmpeg conceals + emits the damaged picture too). Found on a real corrupt
# broadcast capture (3sat HD .ts, 2474/2475). Regression (flush-drain orphan
# recovery) => stall losing the last picture.
cabac_orphan 24
